# 🛠️ Meticulous Fix Plan for Nexus Halo and Flutter App Bugs

This document contains step-by-step code-level plans to resolve the bugs identified in the deep scan.

---

## 📱 Part 1: Android Build & Config (`app/android/`)

### 1. Missing Kotlin Plugin
**File:** `app/android/app/build.gradle.kts`
**Issue:** The build script is missing the Kotlin Android plugin, which is required since `kotlin { ... }` configuration is used and Android apps using Kotlin need it applied explicitly in the app module.
**Fix:** Apply the `org.jetbrains.kotlin.android` plugin inside the `plugins` block.
```diff
 plugins {
     id("com.android.application")
+    id("org.jetbrains.kotlin.android")
     // The Flutter Gradle Plugin must be applied after the Android and Kotlin Gradle plugins.
     id("dev.flutter.flutter-gradle-plugin")
 }
```

### 2. BootReceiver is not Exported
**File:** `app/android/app/src/main/AndroidManifest.xml`
**Issue:** The `BootReceiver` is meant to listen for `BOOT_COMPLETED` from the system, but `android:exported="false"` blocks the Android system from delivering the broadcast intent on Android 12+.
**Fix:** Change the `android:exported` attribute to `true`.
```diff
         <!-- Boot receiver — reiniciar el servicio tras reboot -->
         <receiver
             android:name="com.pravera.flutter_foreground_task.receiver.BootReceiver"
-            android:exported="false">
+            android:exported="true">
             <intent-filter>
                 <action android:name="android.intent.action.BOOT_COMPLETED" />
             </intent-filter>
         </receiver>
```

### 3. Circular Evaluation Dependency
**File:** `app/android/build.gradle.kts`
**Issue:** The `subprojects` block forces *every* subproject (including `:app`) to evaluate `:app` first, causing a circular dependency where `:app` depends on itself.
**Fix:** Filter out the `:app` module before applying the evaluation dependency.
```diff
 subprojects {
-    project.evaluationDependsOn(":app")
+    if (project.name != "app") {
+        project.evaluationDependsOn(":app")
+    }
 }
```

### 4. State Desync / Unresolved Kotlin Reference
**File:** `app/android/gradle.properties`
**Issue:** `android.builtInKotlin=false` is a deprecated/problematic flag from older Flutter templates that causes the IDE and Gradle to desync and lose Kotlin references.
**Fix:** Remove the property entirely so the build uses the standard Kotlin plugin.
```diff
-android.newDsl=false
-# This builtInKotlin flag was added by the Flutter template
-android.builtInKotlin=false
```

### 5. Unhandled Exception Risk (`FileNotFoundException`)
**File:** `app/android/settings.gradle.kts`
**Issue:** Reading `local.properties` via `inputStream()` assumes the file exists. If it doesn't (e.g., fresh clone, CI/CD environment), it crashes the entire build.
**Fix:** Add an `exists()` check before attempting to read it.
```diff
         run {
             val properties = java.util.Properties()
-            file("local.properties").inputStream().use { properties.load(it) }
+            val localProperties = file("local.properties")
+            if (localProperties.exists()) {
+                localProperties.inputStream().use { properties.load(it) }
+            }
             val flutterSdkPath = properties.getProperty("flutter.sdk")
```

### 6. Extreme Memory Over-Allocation
**File:** `app/android/gradle.properties`
**Issue:** Allocating 8GB for Heap and 4GB for Metaspace is extremely high for a Flutter Android build and can crash machines with lower RAM or cause GC pauses.
**Fix:** Normalize JVM arguments to standard bounds.
```diff
-org.gradle.jvmargs=-Xmx8G -XX:MaxMetaspaceSize=4G -XX:ReservedCodeCacheSize=512m -XX:+HeapDumpOnOutOfMemoryError
+org.gradle.jvmargs=-Xmx2048m -XX:MaxMetaspaceSize=512m -XX:+HeapDumpOnOutOfMemoryError
```

**File:** `app/lib/main.dart` (Inside `_bootstrap()`)
**Issue:** When the watch toggles Radar Mode, `onRadarModeChanged` fires, and the app immediately writes the radar state *back* to the watch (`bleService.writeRadarMode`), which triggers another BLE notification, causing an infinite echo loop.
**Fix:** Remove the redundant `writeRadarMode` back to the watch from this callback.
```diff

-        // APP -> BLE: Notificar al reloj si entramos a radar mode para mantenerlo despierto
-        bleService.writeRadarMode(active).catchError((e) => print('[BLE] Radar toggle error: $e'));

```

### 4. State Desync & Silent Bootstrapping Failure
**File:** `app/lib/main.dart` (Inside `_bootstrap()` & `build()`)
**Issue:** If `_bootstrap()` catches an error, it sets `_bootError` but still calls `setState(() => _initialized = true)`. This forces the UI to render `HomeScreen()` with completely uninitialized services (crashing downstream streams and listeners silently). 
**Fix:** Prevent `_initialized = true` on failure and render a direct Retry button on the loading screen instead of bypassing it to show a Snackbar.
```diff

       print('[APP] Stack: $stack');
-      setState(() => _bootError = e.toString());
+      setState(() {
+        _bootError = e.toString();
+        _initialized = false;
+      });
+      return; // Skip setting _initialized = true to block HomeScreen
     }
```
And add a simple retry button inside the `!_initialized` layout so the user isn't stuck:
```diff

                 // ... existing error text ...
+                const SizedBox(height: 16),
+                ElevatedButton(
+                  onPressed: _bootstrap,
+                  child: const Text('Reintentar'),
+                ),
               ],
```

### 1. Critical Architecture Flaw (Main Isolate Suspension)
**File:** `foreground_service.dart`
**Issue:** `flutter_foreground_task` executes its background logic within a separate isolate. The BLE and GPS services run on the main UI isolate, which Android suspends in the background.
**Fix:** Move the initialization and execution lifecycle of the background services into the background isolate via `_ForegroundTaskHandler`.
```dart
class _ForegroundTaskHandler extends TaskHandler {
  // Initialize services here so they run inside the background isolate
  final BleService _bleService = BleService();
  final LocationService _locationService = LocationService();

  @override
  void onStart(DateTime timestamp, SendPort? sendPort) {
    print('[FG] Background Isolate started');
    _bleService.initialize();
    _locationService.start();
  }
  
  @override
  void onDestroy(DateTime timestamp, SendPort? sendPort) {
    _bleService.dispose();
    _locationService.dispose();
  }
}
```

## 🟢 Part 5: Firmware LED Controller & Haptic Motor (`led_controller.cpp`, `haptic.cpp`)

**File:** `led_controller.cpp`
**Issue:** `setPower(false)` sets `power_on = false` right before calling `show()`. Because `show()` immediately aborts if `!power_on`, the blanking frame is never transmitted and the EasyDMA peripheral is bypassed completely.
