# Chat Summary - 2026-07-07 (Final Wrap-up)

## Context
Following earlier architecture fixes for the Nexus Halo project, the user attempted to boot the watch and found it was completely unable to be discovered via Bluetooth. The user also explicitly requested not to use sloppy `delay()` polling methods in the final solution.

## Investigation & Fixes Applied
1. **Bluetooth RTOS Deadlock**:
   - Analyzed the previous power management code in `power.cpp`, which was changed to use raw `__WFE()` to keep the device awake. 
   - Since the Seeed nRF52840 core uses **FreeRTOS** under the hood, calling `__WFE()` directly from the main `loop()` task starved the FreeRTOS scheduler. This caused the Bluetooth stack to deadlock and the device to crash immediately upon booting into deep sleep mode.
   
2. **True FreeRTOS Deep Sleep**:
   - Removed all `delay()` polling and bare-metal `__WFE()` instructions.
   - Implemented the elegant `suspendLoop()` API inside `power.cpp` `enterDeepSleep()`. This cleanly halts the main Arduino task indefinitely, allowing the FreeRTOS scheduler to execute the Idle Task, which natively puts the CPU to sleep via `sd_app_evt_wait()` without disrupting the SoftDevice/BLE.
   - Added `resumeLoop()` to all interrupt handlers (`onMotionWakeup`) and BLE callbacks (`ble_handler.cpp` and `nexus_halo.ino`). This allows the watch to wake up exactly when an interrupt fires or when the app connects, with absolute zero polling overhead.
