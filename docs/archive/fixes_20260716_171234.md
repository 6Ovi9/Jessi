### 4. `wake_calibration_screen.dart`: DB write after BLE — desync if DB fails; no mounted guard before context.read (HIGH)
* **File:** `app/lib/screens/wake_calibration_screen.dart`
* **Lines:** 140-151
* **Explanation:** Writing to BLE before DB means the DB can fail and leave the watch out-of-sync. Additionally, `context.read` is used after `await` without checking `mounted`. Swapping to write to DB first causes a silent state mismatch if BLE then fails. 
* **Fix:** Swap the order to write to DB first, then BLE. Add a `mounted` check. If the BLE write fails, rollback the DB state to the old configuration so the UI accurately reflects the watch state.
```dart
    final repo = context.read<PartnerRepository>();
    final cfg = repo.config ?? WatchConfig.defaultFor(repo.myUserId);
    final newCfg = cfg.copyWith(wakeThreshold: reg);

    try {
      await repo.saveConfig(newCfg);
    } catch (e) {
      if (!mounted) return;
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text('Error DB: $e'), backgroundColor: Colors.red),
      );
      return;
    }

    try {
      await ble.writeWakeThreshold(reg);
    } catch (e) {
      // Rollback on BLE failure
      try {
        await repo.saveConfig(cfg); 
      } catch (_) {}
      if (!mounted) return;
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(content: Text('Error BLE: $e. Reverted.'), backgroundColor: Colors.orange),
      );
    }
```

### 17. `config_json_buf` race in `_onWrite()` BLE callback
*   **File**: `nexus_halo/ble_handler.cpp`
*   **Lines**: 280–288 (`_onWrite`, `config_char` branch)
*   **Explanation**: The BLE write callback `_onWrite()` uses `noInterrupts()` / `interrupts()` (lines 283–286) to protect writes to `config_json_buf`. On the nRF52 FreeRTOS port, `noInterrupts()` only masks hardware IRQs — it does NOT prevent the FreeRTOS scheduler from switching threads. The main loop can still be preempted mid-parse. Additionally, if `data` is ever `nullptr` or `len == 0` (defensive case), `memcpy(config_json_buf, nullptr, 0)` is undefined behavior.
*   **Important — Two separate functions:** `_onWrite()` is the receive path (BLE thread → buffer). `notifyConfig()` (lines 375–385) is the transmit path called from the MAIN LOOP only — it already has a null guard (`if (!json) return`) and does NOT need mutex protection. Only `_onWrite` is the race.
*   **Fix Code**: Replace `noInterrupts()` / `interrupts()` in `_onWrite()` with a FreeRTOS Mutex. Initialize it in `BLEHandler::begin()`. The main loop must also take the mutex before calling `getConfigJson()` or parsing the buffer.
    ```cpp
    // In BLEHandler::begin() — create mutex once:
    json_mutex = xSemaphoreCreateMutex();

    // In _onWrite(), config_char branch — replace noInterrupts/interrupts:
    if (len == 0 || !data || len >= sizeof(config_json_buf)) return;
    if (xSemaphoreTake(json_mutex, portMAX_DELAY)) {
        uint16_t copy_len = len < sizeof(config_json_buf) - 1 ? len : sizeof(config_json_buf) - 1;
    ```

### 18. WAKING_UP transitions unconditionally every tick
*   **File**: `nexus_halo/nexus_halo.ino`
*   **Lines**: 985-996
*   **Explanation**: The transition to `CLOCK_CONNECTED` or `CLOCK_DISCONNECTED` fires immediately on every loop tick without a timeout, cutting the wake animation off instantly. After a transition inside a state machine case block, a `break;` is needed to avoid fall-through execution.
*   **Fix Code**: Explicitly rely on a timer to ensure the animation finishes, and `break` after transition:
    ```cpp
    if (is_entry) state_machine.resetTimer(1500); // 1.5s animation
    if (state_machine.isTimerExpired()) {
        state_machine.transitionTo(ble_handler.isConnected() ? STATE_CLOCK_CONNECTED : STATE_CLOCK_DISCONNECTED);
        break;
    }
    ```
