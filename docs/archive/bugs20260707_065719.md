# Nexus Halo Bug Report - 2026-07-07

## Firmware Lockup & Deep Sleep Issues
The primary issue reported by the user ("watch is killing itself after going to low power") has been thoroughly investigated by our subagent team.

### Bug 1: System OFF Deep Sleep Lockup
**Component**: `nexus_halo/power.cpp`
**Description**: The `enterDeepSleep` function was recently changed to use `sd_power_system_off()`. The nRF52840's `System OFF` mode disables the USB peripheral, triggers a full system reset on wake, and prevents standard `attachInterrupt` GPIO events from waking the device. The current firmware lacks `SENSE` configuration (e.g., `nrf_gpio_cfg_sense_input`) required for wake-up from System OFF, and our architecture relies on maintaining state across sleep intervals.
**Impact**: The watch completely powers down the main CPU and USB, appearing dead. It cannot be woken by gestures, only by power cycling (unplugging and replugging USB/battery).
**Fix required**: Revert `power.cpp` lines 193-206 to use `System ON` mode via `sd_app_evt_wait()` (if SoftDevice enabled) or `__WFE()` (if not).

### Bug 2: I2C/TWI Internal Bus Deadlock
**Component**: `nexus_halo/nexus_halo.ino`
**Description**: A workaround for an Mbed OS `Wire.end()` bug was manually clearing the `ENABLE` and `PSEL` registers for `NRF_TWIM0` and `NRF_TWIM1`/`NRF_TWI1`. However, `Wire1` (mapped to `TWIM1`) is used by the internal LSM6DS3 IMU. Manually forcing the hardware registers off underneath the mbed OS driver for `Wire1` causes a hard fault/deadlock when the IMU is accessed during or after wake-up.
**Impact**: Contributes to severe crashes and lockups.
**Fix required**: Remove the manual register modifications for `NRF_TWIM1` and `NRF_TWI1` in the `nexus_halo.ino` setup logic. Only clear `TWIM0`/`TWI0`.

### Bug 3: Haptic Controller Event Loop Starvation
**Component**: `nexus_halo/haptic.cpp`
**Description**: The `playPattern` function uses a blocking `for` loop with `delay()`. This freezes the main event loop for up to ~1000ms.
**Impact**: Starves BLE handlers and background tasks, which can cause severe lockups, BLE disconnects, or watchdog resets.
**Fix required**: Revert `playPattern` to simply configure the state machine variables so `update()` handles it non-blockingly.

### Bug 4: Radar Mode Battery Drain & BLE Spambot
**Component**: `nexus_halo/state_machine.cpp`
**Description**: When `STATE_RADAR_MODE` times out, it directly calls `transitionTo(STATE_DEEP_SLEEP)`. Because the UI exit gesture is bypassed, `ble_handler.notifyRadarModeActive(false)` is never sent to the phone.
**Impact**: The phone continues to aggressively stream compass data at 50Hz, constantly waking the watch from sleep via BLE interrupts and causing massive battery drain.
**Fix required**: Change the timeout behavior in `state_machine.cpp` to transition back to the clock mode, which will then handle its own idle timeout gracefully.

### Note on LEDs
The LED logic in `led_controller.cpp` (`showClock`) was analyzed. The math for time carry-over and counter-clockwise LED mapping handles modulo wrapping correctly and is safe from buffer overflows. The lack of interpolation between hours is a design choice (snapping exactly to the integer hour), which is stable, though it diverges slightly from previous smooth interpolation behavior.
