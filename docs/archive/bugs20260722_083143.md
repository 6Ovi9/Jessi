# Communication & Sync Bug Audit Report — 2026-07-22 08:31:43

This document contains the consolidated findings, root cause analysis, and recommended fixes for watch-to-app and app-to-watch communication issues across the entire stack (Firmware, Flutter App BLE Service, Native Android Background Engine, and Supabase Backend).

---

## Executive Summary & Database Question Answer

### Question: "Do you think dropping the current database could help?"
**Answer:** **NO, dropping the database will NOT help and is NOT recommended.**

**Rationale:**
1. **Schema Integrity:** The `haptic_events`, `locations`, and `watch_config` tables and indices in PostgreSQL are completely valid.
2. **True Root Cause:** The failure of `"Probar Vibración"` to appear in `haptic_events` is caused by **Row Level Security (RLS) policies** rejecting unauthenticated (`anon`) connections in Supabase (`init-db.sql`), as well as client-side user role initialization timing in `SyncService`.
3. **Data Loss:** Dropping the database would wipe active pairing codes, tokens, and stored user preferences without resolving the RLS policy restriction or authentication token headers.
4. **Correct Action:** Executing SQL to grant RLS permissions (`TO anon, authenticated`) resolves the issue cleanly without dropping data.

## Comprehensive Bug Audit

### Category 1: Watch Firmware (`nexus_halo/`)

#### 1.1 Immediate `saveToFlash()` inside `updateFromJson()` Defeating 3-Second Deferred Timer
- **Location**: `nexus_halo/runtime_config.cpp:222`
- **Root Cause**: `updateFromJson()` explicitly calls `saveToFlash()` immediately after parsing JSON. Flash writes take **80–100 ms** of synchronous CPU execution time. Since the mobile app sends 5 JSON packets at **150 ms intervals**, the watch CPU blocks writing Flash for packet 1 and fails to process or drops subsequent incoming BLE packets containing needle colors (`chc`, `chd`), haptic colors (`ctx`, `crx`), and brightness (`br`).
- **Fix**: Remove `saveToFlash()` from `updateFromJson()` (line 222). Rely entirely on the 3-second deferred timer in `nexus_halo.ino` (`config_save_pending`).

#### 1.2 Mutex Violation in `notifyConfig()` causing Data Races on `config_json_buf`
- **Location**: `nexus_halo/ble_handler.cpp:408–419`
- **Root Cause**: `notifyConfig()` writes directly to `config_json_buf` using `memcpy` without acquiring `json_mutex`.
- **Fix**: Protect `config_json_buf` in `notifyConfig()` with `xSemaphoreTake(json_mutex, portMAX_DELAY)`.

#### 1.3 BLE Echo Loop / Notification Feedback Storm
- **Location**: `nexus_halo/nexus_halo.ino:477`
- **Root Cause**: `ble_handler.notifyConfig(local_json)` is called immediately after receiving a config JSON from the mobile app via BLE, causing an endless write-notify-write feedback loop.
- **Fix**: Only call `notifyConfig()` when the config is modified locally on the watch (e.g. triple-flick brightness reset), NOT in response to incoming BLE writes from the app.

### Category 2: Flutter App BLE & Background Execution Engine (`app/`)

#### 2.1 GATT ATT MTU Overflow in Chunked JSON Config Transmission
- **Location**: `app/lib/services/background_engine.dart:440–516`
- **Root Cause**: `_writeConfigInChunks` assumes chunk JSON payloads up to 120 bytes are safe. If MTU negotiation fails or defaults to 23 bytes (20-byte data payload), `_configChar.write()` throws an MTU error and drops the config chunk silently.
- **Fix**: Check `FlutterBluePlus.currentMtu` dynamically or send 20-byte safe payload chunks.

#### 2.2 Unqueued Cumulative Transmission Delays & Out-of-Order Config Corruption
- **Location**: `app/lib/services/background_engine.dart:440–516`
- **Root Cause**: Rapidly adjusting UI sliders triggers concurrent `_writeConfigInChunks` executions. The 5 chunks of execution A and execution B interleave on the BLE GATT socket, sending scrambled JSON payloads to the watch.
- **Fix**: Implement a FIFO write queue (`Queue<String> _configWriteQueue`) to ensure only one config transmission cycle runs at a time.

#### 2.3 Color Hex Formatting Inconsistency
- **Location**: `app/lib/screens/settings_screen.dart:985–989`
- **Root Cause**: `onSelected('FF${hex.substring(2)}')` assumes the first 2 characters of `toRadixString(16).padLeft(8, '0')` are alpha, but when RGB values have leading zeroes, `.substring(2)` slices color channel bytes instead.
- **Fix**: Use exact bitwise masks: `(pickerColor.value & 0x00FFFFFF).toRadixString(16).padLeft(6, '0')`.

---

### Category 3: Supabase Backend & App Sync (`backend/` & `app/`)

#### 3.1 RLS Policy Restriction rejecting `anonKey` connections
- **Location**: `backend/init-db.sql:260–274`
- **Root Cause**: RLS policies on `haptic_events`, `locations`, and `watch_config` are set `TO authenticated` ONLY. Clients connecting via PostgREST with `anonKey` without active auth session connect as `anon` role and are rejected with HTTP 401 / 42501 (Permission Denied).
- **Fix**: Grant RLS policies `TO anon, authenticated` on `haptic_events`, `locations`, and `watch_config`.

#### 3.2 Default User Role Timing Mismatch in `SyncService`
- **Location**: `app/lib/services/sync_service.dart:70`
- **Root Cause**: `_myUserId` defaults to `'A'`. If "Probar Vibración" is clicked before `initialize(userId)` resolves, `from_user` is inserted as `'A'` and `to_user` as `'B'`.
- **Fix**: Ensure `SyncService` verifies `_myUserId` is initialized before sending events.

## Action Plan Summary

1. **Firmware (`nexus_halo`):**
   - Comment out `saveToFlash()` in `runtime_config.cpp` (Line 222).
   - Add mutex protection in `ble_handler.cpp` (`notifyConfig`).
   - Remove redundant `notifyConfig()` call on incoming BLE writes in `nexus_halo.ino`.

2. **Backend (`backend/init-db.sql`):**
   - Apply updated RLS policies allowing `TO anon, authenticated` on `haptic_events`, `locations`, and `watch_config`.

3. **Flutter App (`app/`):**
   - Implement bitwise hex masking in `settings_screen.dart`.
   - Validate user role and error handling in `sync_service.dart`.
