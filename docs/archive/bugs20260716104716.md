# Jessi Codebase Review - 2026-07-16 10:47:16
**Status**: Completed. All findings have been double-verified by a Primary and Secondary Agent in each domain across two comprehensive sweeps.

## 1. Firmware (`nexus_halo` C++) - 7 Verified Issues

1.  **Unpowered LED Calls in Deep Sleep (Logic / Architecture Error)**
    *   **Description**: In `handleStateDeepSleep`, the logic attempts to pulse the LED for low battery warnings. However, `led_controller.setPower(false)` is explicitly called on sleep entry. The `errorBattery()` function has a guard (`if (!power_on) return;`), meaning the pulse will instantly return and never illuminate the LED. If bypassed, there is no subsequent `delay()` and `setPower(false)`, meaning the LED ring would remain powered indefinitely while the CPU sleeps, draining the battery.
2.  **Blocking Delays in Haptic TX State (Architecture Issue)**
    *   **Description**: In `handleStateHapticTX`, the state function is executing sequential `delay(100)` calls to pulse the LED. Because this occurs directly in the main `loop()` via the state machine, it creates a synchronous 300ms busy-wait. This violates the non-blocking architecture and will stall BLE updates and IMU readings.
3.  **Time Zone Offset Applied to Unsynced Clock (Logic Error)**
    *   **Description**: In `handleStateClockConnected` and `Disconnected`, a negative timezone offset is applied. When `!time_synced`, `current_unix_ms` returns device uptime (`millis()`). Applying a timezone offset (e.g., -3600 seconds) to the bare device uptime correctly locks the display to `00:00:00` for the first full hour after boot. If the clock is not synced, it should simply display uptime without applying the timezone offset.
4.  **Triple Flick Sequence Timeout Mismatch (Logic Error)**
    *   **Description**: In `gesture.cpp`, the gesture was updated from Double Flick to Triple Flick, but the global sequence timeout (`double_flick_window` of 800ms) was not adjusted. The 800ms window applies to the *entire sequence*. With a ~280ms cooldown between flicks, the user has only a ~240ms window to execute the 3rd flick before the sequence preemptively times out and falls back to `GESTURE_TAP_DOUBLE`.
5.  **`delay(10)` in `enterDeepSleep` Prevents True Low-Power Mode (Architecture Issue)**
    *   **Description**: When `SOFTDEVICE_PRESENT` is true, `PowerManager::enterDeepSleep()` uses a simple `delay(10)`. Since `delay()` relies on the FreeRTOS SysTick timer, the CPU will repeatedly wake up to service the timer interrupt instead of properly halting execution with `sd_app_evt_wait()`, causing severe battery drain during "deep sleep".
6.  **Radar Mode Timeout Leaves App in Rapid Streaming Mode (Logic Error)**
    *   **Description**: When the watch times out of Radar Mode, `StateMachine::update()` calls `transitionTo(previous_state)`. Because `_exitState()` is completely empty, the cleanup command (`notifyRadarModeActive(false)`) is skipped, leaving the mobile app rapidly streaming GPS/BLE data and draining the phone's battery.
7.  **BLE Wakeup Race Condition Bypasses IMU/Compass Initialization (Logic Error)**
    *   **Description**: BLE command handlers explicitly chain state transitions (e.g., `transitionTo(STATE_WAKING_UP)` immediately followed by `transitionTo(STATE_COMPASS_CALIBRATION)`). This bypasses the execution of `handleStateWakingUp()`. Essential hardware wakeup routines (like `lsm6ds3.begin()`, compass power up, and LED power) are completely skipped, silently corrupting calibration data.

## 2. Flutter App - 9 Verified Issues

1.  **Missing "Phase 2" Features (Hallucinated / Lost Code)**
    *   **Description**: Previous documentation claims that `WatchConfig` was updated to include `colorHapticTx`, `colorHapticRx`, etc., and that a `"Colores y Brillo de Toques"` UI was added to `settings_screen.dart`. These changes do not exist in the codebase. The `pubspec.yaml` version remains `1.1.0+2`. This is hallucinated or overwritten code.
2.  **Missing BLE Auto-Reconnect on Disconnect (Logic Error)**
    *   **Description**: In `ble_service.dart`, standard disconnects trigger `_onDisconnected()`, which cleans up subscriptions but *never schedules a reconnect*. The auto-reconnect logic exists exclusively inside the `onError` block. Standard disconnects will never recover until the app is restarted.
3.  **Infinite Reconnect Loop / Broken Exponential Backoff (Logic Error)**
    *   **Description**: `connectToDevice()` unconditionally sets `_reconnectAttempt = 0` at the very beginning. When the `_retryTimer` fires, it calls `connectToDevice()`, clearing the attempt counter every single time. This entirely circumvents the `_maxReconnectAttempts` limit and breaks the exponential backoff, causing an infinite loop of retries at a constant 5-second interval.
4.  **Unreachable Error Snackbar (Dead Code)**
    *   **Description**: In `main.dart`, logic exists to show a Snackbar when `_bootError != null`. However, if `_bootError` is set, `_initialized` is guaranteed to be `false`, triggering an early return. The Snackbar logic is dead code and will never be executed.
5.  **Foreground Service Execution Model Mismatch (Architecture Flaw)**
    *   **Description**: `foreground_service.dart` initializes a background handler, but all actual business logic (`LocationService`, `BleService` timers) remains on the main UI isolate. When the app is backgrounded, Android aggressively suspends the main isolate, meaning GPS polling and BLE syncing silently pause despite the foreground service running.
6.  **Time Sync Retry Limit Broken (Logic Error)**
    *   **Description**: `BleService.syncTime` has an early return `if (attempt > 3)`. When `attempt = 3`, a failure schedules `syncTime(4)`, which instantly triggers the early return and silently aborts. The `else` block (which schedules a 15-minute fallback timer) is never reached.
7.  **GPS Polling Race Condition on Radar Mode (Race Condition)**
    *   **Description**: `LocationService.setRadarModeActive(true)` calls `_doGpsUpdate()` (unawaited) and instantly calls `_scheduleNextUpdate()`. This starts the next polling timer immediately, allowing concurrent, overlapping calls to `Geolocator.getCurrentPosition`, which can crash the location stack.
8.  **BLE MTU & Subscription Initialization Race Condition (Race Condition)**
    *   **Description**: `BleService._onConnected` uses `await Future.delayed(500ms)` before requesting MTU and subscribing. If the device disconnects during this 500ms window, the code resumes and executes the BLE requests against a disconnected device, causing bridge exceptions.
9.  **Location Services Disabled Fails Silently (UI Flaw)**
    *   **Description**: In `LocationService.start()`, if `checkAndRequestPermissions()` returns false (e.g., GPS is disabled in OS), it prints to the console and aborts. It does not update state flags or notify the UI, leaving the app stuck on "Iniciando servicios..." indefinitely.

## 3. Backend & Database - 10 Verified Issues (ALL FIXED)

1.  **Initial Location Timestamps Drop Early Updates (HIGH)**
    *   **Description**: The initial default rows for `locations` are seeded using `DEFAULT NOW()`. If a client device's clock is slightly behind the server, its first GPS updates will have `NEW.updated_at <= OLD.updated_at` and be dropped by `enforce_location_order()`.
2.  **Nginx Realtime WebSocket Routing Stripping Prefix (HIGH)**
    *   **Description**: The `location /realtime/v1/` block uses `rewrite ^/realtime/v1/(.*)$ /$1 break;`, forwarding requests as just `/websocket`. Supabase Realtime natively expects the full path (or `/socket/websocket`). Stripping the prefix will likely result in a `404 Not Found` from the Phoenix server, breaking real-time syncing.
3.  **Changing `JWT_SECRET` Breaks Hardcoded Keys (HIGH)**
    *   **Description**: The `.env` file advises the user to change `JWT_SECRET`. However, `ANON_KEY` and `SERVICE_KEY` are provided as hardcoded JWTs signed with the *default* secret. Changing the `JWT_SECRET` without manually generating new JWTs for these keys will cause all client API requests to fail signature validation.
4.  **RLS Policy Names Contradict Implementation (LOW)**
    *   **Description**: Policies like `"Authenticated users can upsert own location"` use `FOR ALL TO authenticated USING (true) WITH CHECK (true)`, allowing any authenticated user to read/update/delete any user's data. This contradicts the policy name.
5.  **Missing Dependency in Docker Compose (LOW)**
    *   **Description**: The `studio` service depends on `meta:8080` (via `STUDIO_PG_META_URL`), but `meta` is missing from the `depends_on` list under the `studio` service, which could cause transient startup errors.
6.  **Location Updates Silently Dropped Due to Missing Timestamp Trigger (HALLUCINATED BUG - FIXED BY REMOVAL)**
    *   **Description**: The trigger to auto-update the timestamp (`trg_locations_updated_at`) was originally thought to be missing. *Correction*: It was omitted intentionally! Re-adding it forces the DB to overwrite the client's timestamp with the server's `NOW()`, causing `enforce_location_order` to drop valid updates due to client-server clock drift. The fix was to ensure this trigger is permanently removed.
7.  **Auth API Base URL Routes Emails to a Blank Gateway Page (HIGH)**
    *   **Description**: The `auth` service uses `API_EXTERNAL_URL: ${API_EXTERNAL_URL:-http://localhost:8000}`. GoTrue uses this for magic links. However, the Nginx reverse proxy requires the `/auth/v1/` prefix. Users clicking email links will hit Nginx's default root path and receive a dummy JSON response.
8.  **Database Schema Missing Firmware "Phase 2" Config Fields (HIGH)**
    *   **Description**: The `watch_config` table is completely missing the "Phase 2" Haptic fields (like `color_haptic_tx`, `brightness_haptic_tx`). Attempts from Flutter to sync these properties will cause PostgREST schema validation errors.
9.  **Missing Auth Redirect Allow-List & Incorrect Site URL (HIGH)**
    *   **Description**: `GOTRUE_SITE_URL` defaults to `http://localhost:3000` and `GOTRUE_URI_ALLOW_LIST` is empty (`""`). This will block any Flutter app deep-link redirects (e.g., `jessi://`) after authentication.
10. **Orphaned Database Maintenance Function (MEDIUM)**
    *   **Description**: `cleanup_old_haptic_events()` is defined, but there is no `pg_cron` schedule, trigger, or worker to ever execute it. It is dead code and the `haptic_events` table will grow infinitely.
11. **Missing SMTP Configuration for GoTrue (MEDIUM)**
    *   **Description**: `GOTRUE_EXTERNAL_EMAIL_ENABLED: "true"` is set, but no `GOTRUE_SMTP_*` variables are defined. Password resets and magic links will crash.
