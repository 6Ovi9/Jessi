# Nexus Halo & App Bug Report
*Generated: 2026-07-06T10:28:25*

## Firmware Bugs & Functional Issues

### Power Management (`nexus_halo/power.cpp` & `power.h`)
1. **PDM Microphone Power Drain (`PDM.end()` API Issue)**: Calling `PDM.end()` and disconnecting GPIO pins on the nRF52840 is insufficient to stop the PDM peripheral completely. The Arduino core implementation often leaves the High-Frequency Clock (HFCLK) or EasyDMA active. Fix: Manually disable the PDM peripheral registers.
2. **LSM6DS3 Wake-Up Interrupt Hardware Limitation**: Attempts to clear a "latched" wake-up flag by reading `WAKE_UP_SRC`. However, the Activity/Wake-up interrupt on the LSM6DS3 *cannot* be latched. It asserts for exactly `1/ODR_XL` and then de-asserts. Use edge-triggered interrupts (e.g., `RISING`).
3. **XIAO nRF52840 Battery Pin Hardware Risk**: The code drives `PIN_VBAT_ENABLE` `HIGH` (as `OUTPUT`) to disable the voltage divider. This can cause conflicts and risk damaging the pin. Configure as `INPUT` (high impedance) instead.
4. **`sd_app_evt_wait()` Deep Sleep Misunderstanding**: `sd_app_evt_wait()` only enters System ON Idle mode. Background timers or unhandled pending interrupts cause it to return immediately. Use `sd_power_system_off()` for true deep sleep.

### Firmware Configuration (`nexus_halo/config.h`)
1. **SK6812 NeoPixel & BLE Disconnections**: Driving SK6812 LEDs using standard bit-banged NeoPixel libraries disables interrupts, causing the nRF52840's BLE stack to drop connections. Use a hardware-accelerated DMA driver.
2. **LSM6DS3TR-C vs LSM6DS3 Interrupt Configuration**: Standard LSM6DS3 libraries fail to trigger interrupts on the TR-C variant because it requires an additional `INTERRUPTS_ENABLE` flag to be set in the `TAP_CFG` register.
3. **IMU Wake Duration Timing Error**: `IMU_ACCEL_RATE_HZ` is set to `12.5Hz`, but comments for `IMU_WAKE_UP_DUR` assume `26Hz` rate. This configuration logic bug makes the rise-to-wake gesture sluggish.
4. **Invalid LIS3MDL Compass ODR**: `COMPASS_UPDATE_RATE_HZ` is set to `30`. The LIS3MDL hardware does not support a 30Hz Output Data Rate.
5. **BLE Sleep Advertising Interval**: `BLE_ADVERTISING_INTERVAL_MS_SLEEP` is set to `1600` ms. (Note: Originally flagged for Apple compliance, but valid for Android).
6. **Struct Padding over BLE**: `BLE_LocationData` and `WatchConfig` lack `__attribute__((packed))`, causing alignment mismatches.
7. **BLE Characteristic UUID Overlap**: Custom UUIDs overlap with Bluetooth SIG assigned 16-bit numbers (like `0x2A58`) and will be ignored by iOS.
8. **Firmware Clock Sync Logs UTC Instead of Local Time**: The firmware properly handles UTC + TZ offset to display local time on the LEDs, but prints the raw UTC time to the serial monitor without adding the offset, causing developer confusion.

### LED Controller (`nexus_halo/led_controller.cpp` & `led_controller.h`)
1. **Critical Power-Off Blanking Bug**: `setPower(false)` sets `power_on = false` before calling `show()`. But `show()` immediately aborts if `!power_on`. The blackout frame DMA transfer is skipped entirely, abruptly cutting power and leaving LEDs in a floating/flashing state.
2. **Clock Display Desync & Millisecond Truncation**: The `seconds` hand advances dynamically, but `minutes` and `hours` are static if updates stop, causing desync. Also, `showClock()` ignores `current_millis` offset, causing the seconds hand to natively lag up to 999ms depending on when the sync packet arrives.
3. **Hardcoded Magic Numbers vs `LED_COUNT`**: `LED_COUNT` is correctly defined in headers, but the value `12` is dangerously hardcoded in loops and visual math throughout the cpp file, breaking compatibility if `LED_COUNT` changes.
4. **RGBW Protocol Desync (CRITICAL)**: Code assumes the SK6812MINI-E is a standard 24-bit GRB LED. It is actually a 32-bit GRBW LED. Writing 24-bit packets to a 32-bit strip causes complete color chaos as the bits misalign across the strip.
5. **Seconds Hand Going Backwards (Hardware LED Wiring)**: The physical LED ring is wired counter-clockwise. Mapping a clockwise logical index directly to the physical LED index makes the clock hands rotate backwards in reality.

### IMU Calibrator (`nexus_halo/imu_calibrator.cpp` & `imu_calibrator.h`)
1. **WAKE_UP_THS Calculation Error (Critical Bug)**: The code converts threshold using `threshold_mg / 62.5f`, but at ±2g, `WAKE_UP_THS` resolution is 31.25 mg/LSB. The physical wake-up threshold will be twice as sensitive as intended, causing frequent false-positive wakeups.
2. **Flawed Gesture Debounce/Termination Logic**: The logic forces the gesture to stay alive for the *first 200ms*. After 200ms, a single sample dropping below half the threshold instantly terminates the gesture, prematurely splitting movements and recording incorrect max acceleration.
3. **Insufficient Polling Rate**: Throttling IMU polling to exactly 10Hz without using the IMU's hardware FIFO means the software will miss the true maximum acceleration of rapid wrist gestures.
4. **Unit Documentation Mismatch**: The comment states `// Magnitude in m/s²`, but `readFloatAccelX/Y/Z()` return values in `g`.
5. **Dead Variable**: `last_accel_magnitude` is updated but never utilized in gesture detection logic.

### Runtime Configuration (`nexus_halo/runtime_config.cpp` & `runtime_config.h`)
1. **BLE Settings Not Saved**: `updateFromJson` parses BLE JSON but never calls `saveToFlash()`. Config changes made by the user via the app are lost on restart.
2. **OTA Updates Wipe User Settings**: Struct timeout fields were bumped from `uint8_t` to `uint16_t` (40 to 44 bytes). The strict size check in `loadFromFlash()` immediately fails for older files before checking magic version, reverting the device to factory defaults after an OTA update.
3. **File System Atomicity Broken**: Calling `InternalFS.remove(RUNTIME_CONFIG_FILE)` immediately before `InternalFS.rename(...)` breaks LittleFS's natural atomicity. Power loss between these calls loses both old and new configs.
4. **Fragile JSON Parser**: The custom JSON parser checks `prev == ' '` which can falsely trigger if unrelated string values contain a key substring, potentially corrupting settings.
5. **Silent Type Wraparound & Missing Bounds Checks**: Zero validation on values parsed by `_findJsonInt()`. Negative values can wrap around in `uint8_t` or `uint16_t` (e.g., `-10` becomes 246% brightness).

### BLE Handler (`nexus_halo/ble_handler.cpp` & `ble_handler.h`)
1. **BLE Connection Parameter Update Timing and Values**: Requesting parameter updates instantly upon connection causes the SoftDevice queue to block, leading to dropped connections. Wait 500ms before requesting. Also, requesting 100ms violates Apple's multiple-of-15ms rule.
2. **Characteristic Data Truncation due to MTU Limitations**: `setMaxLen(sizeof(config_json_buf) - 1)` does not negotiate MTU, leaving the default 20-byte limit. Any JSON string over 20 bytes will truncate/fail. Fix: Enable MTU negotiation via `configPrphBandwidth`.

### EEPROM Manager (`nexus_halo/eeprom_manager.cpp` & `eeprom_manager.h`)
1. **Defeated Atomicity in File Rename**: Calling `InternalFS.remove("calib.dat")` immediately before `rename` defeats the native atomicity, risking complete data loss during power failure.
2. **Meaningless Timestamp Tracking**: The timestamp is set using `millis() / 1000` which tracks uptime, not absolute time. Reboots reset the timer, breaking chronological checks.
3. **Checksum Implementation Mismatch**: `_calculateChecksum()` uses an additive sum, but integration spec `COMPILATION_VALIDATION.md` explicitly calls for an "XOR checksum", which will break external validation.

### Haptic Motor (`nexus_halo/haptic.cpp` & `haptic.h`)
1. **Major API Documentation Discrepancy (Stuck Motor Bug)**: `playPattern()` is non-blocking despite documentation. Failing to frequently call `update()` causes the motor to be stuck ON indefinitely.
2. **PWM Control is Stubbed (Intensity Bug)**: `setMotor(uint8_t intensity)` bypasses PWM completely, using `digitalWrite(HIGH/LOW)`, making any non-zero intensity run at 100%. Use `analogWrite()`.
3. **Hardware Integration Risks (MOSFET)**: Driving an LRA requires AC (this uses DC MOSFET). If ERM, missing flyback diode causes inductive kickbacks that reset the MCU. Floating gate on boot could cause random activation.
4. **Edge Case Custom Patterns Missing Terminator**: Patterns lacking `{0, 0}` terminator cause out-of-bounds memory reads, producing erratic ghost vibrations.

### State Machine (`nexus_halo/state_machine.cpp` & `state_machine.h`)
1. **Calibration Mode Stuck State (Timer Deadlock)**: When calibration timer expires, `update()` clears the timer but performs no transition. The firmware hangs on the success screen permanently.
2. **Radar/Distance Mode Blind Transition (Visual Glitch)**: Expiration of inactivity timer blindly transitions to `STATE_CLOCK_CONNECTED`, causing a visual bounce glitch if BLE is actually disconnected.
3. **Parameter Shadowing Anti-Pattern**: `update(uint32_t now_ms)` immediately overwrites `now_ms = millis()`, silently discarding the parameter.

### Gesture Detection (`nexus_halo/gesture.cpp` & `gesture.h`)
1. **Dead Code & Logic Flaw in Timeout**: An `else` block checking for gap length is mathematically unreachable because `flick_count` is reset earlier if the time difference exceeds the window. Also uses sliding window instead of strict sequence timeout window.
2. **`reset()` Preserves `last_flick_ms`**: Doesn't reset `last_flick_ms`, causing ignored gestures due to debounce cooldown if the user flicks right after entering a new visual state.
3. **Slow Single Flicks As Double Taps**: A slow single wrist flick (forward swing then slow return swing) can register the return swing after the debounce cooldown, falsely detecting it as a double tap.

### Compass (`nexus_halo/compass.cpp` & `compass.h`)
1. **Blocking File I/O During I2C Error Recovery**: If the I2C bus gets stuck, `reinitBus()` calls `begin()`, which invokes `loadCalibration()`. This performs synchronous flash memory reads inside the RTOS/UI loop, causing 50-100ms hangs.
2. **Calibration Validation Flaw**: Calibration clamps `scale_x` to 1.0f if the watch is not rotated (max=min), which results in `(x - offset) / scale` outputting exactly `0.0` continuously. Needs a minimum magnetic delta check.
3. **Floating I2C Lines During Power-Down**: Disabling pull-ups during sleep leaves I2C lines electrically floating, causing shoot-through currents or corrupted I2C state machine.
4. **Disabled Block Data Update (BDU)**: `CTRL5` initialized to `0x00`. BDU should be enabled to prevent MSB and LSB of an axis from being fetched from two different samples during an ongoing update, preventing corrupted spikes.
5. **Filter Initialization Logic Glitch**: The initial seed for the low-pass filter is unconditionally overridden by the raw instantaneous heading of the second sample due to improper indexing checks.

### Main Firmware Entry (`nexus_halo/nexus_halo.ino`)
1. **`Wire.end()` Hardware Release Bug**: On the Mbed OS core, `Wire.end()` does not properly release the pins from the TWI hardware peripheral. This silently breaks the bit-bang I2C taking over those exact pins.
2. **`sd_app_evt_wait()` / `__WFE()` System Hang**: Calling this manually inside an Mbed OS thread conflicts with Mbed's RTOS scheduler and tickless power management, causing hangs. `delay()` or `yield()` should be used instead.
3. **BLE Calibration Cancel State Jump**: Canceling IMU calibration unconditionally forces transition to `STATE_CLOCK_CONNECTED`, ignoring what mode the user was previously in (e.g. Radar).
4. **Compass Calibration Provides No Visual Feedback**: Starts silently in the background, never transitioning the main state machine, leaving the user with no visual LED cue to rotate the watch.
5. **Time Sync Wrap-Around**: Unsigned calculation `millis() - local_millis` handles 32-bit short-term rollovers, but wraps around entirely if the device stays on without BLE sync for ~49.7 days, causing the clock to jump back.

### Firmware Overall Architecture Notes
1. **IMU Library Gotcha**: Must use `Seeed_Arduino_LSM6DS3` and explicitly initialize `Wire1`, otherwise the IMU completely fails on XIAO nRF52840.

## App Bugs & Functional Issues (Flutter)

### Android Build & Config (`app/android/`)
1. **Missing Kotlin Plugin**: `app/build.gradle.kts` omits `id("kotlin-android")` plugin, breaking the Flutter Gradle Plugin logic.
2. **BootReceiver is not Exported**: `BootReceiver` is set to `android:exported="false"`, which prevents it from receiving `BOOT_COMPLETED` on Android 12+, preventing background tasks from restarting on boot.
3. **Circular Evaluation Dependency**: `settings.gradle.kts` uses declarative loading, but `build.gradle.kts` uses legacy `evaluationDependsOn(":app")` causing an infinite loop / build exception.

### App Initialization (`app/lib/main.dart`)
1. **Initialization Short-Circuit**: `partnerRepo.initialize()` and `syncService.initialize()` are in the same `try...catch`. If `partnerRepo` times out, `syncService` is skipped entirely, permanently breaking location syncing and haptic events.
2. **Memory Leak and Duplicate Polling**: `partnerRepo.addListener(updateLocationIntervals)` appends a listener every time `_bootstrap()` runs. This duplicates polling execution.
3. **Infinite Loop Risk in Radar Mode**: In BLE callbacks, the app responds to `onRadarModeChanged` by writing the state back to the watch (`writeRadarMode(active)`), causing a potential infinite loop of BLE messages.

### UI & Widgets (`app/lib/widgets/watch_preview_widget.dart`)
1. **Reverse Rendering of Clock and Distance**: Calculated using a physical hardware mapping, drawing the clock backwards (e.g., 3 o'clock drawn on the left side).
2. **Off-By-One Logic Error in Distance Mode**: In `_drawDistanceMode`, the 12th LED (`i = 11`) gets `0.0` brightness at max distance. When `normalized = 0.0`, the 1st LED gets `0.0` brightness.
3. **Missing Color Composition in Clock Mode**: When the seconds hand passes over an hour/minute mark, it completely overwrites the color, dimming the time mark due to gamma interpolation.
4. **Overlapped Second Positional Logic**: Fallback logic leaves `frac <= 0.02` unassigned, turning the LED off when interpolation is near `0.0`.

### Screens (`app/lib/screens/`)
1. **Absolute vs Relative Bearing Logic Error**: In `home_screen.dart`, `_buildDistanceCard` rotates the navigation arrow using absolute bearing, ignoring the phone's compass heading. Needs `(bearingToDestination - phoneHeading + 360) % 360`.
2. **State Sync/Overwrite Bug**: `SettingsScreen` uses a detached local `_config`. Sub-screens (Calibration) save to the repo correctly, but saving in `SettingsScreen` overwrites with the stale local copy, erasing calibrations.
3. **BLE Scan & Connection Instability**: In `pairing_screen.dart`, `bleService.stopScan()` is never called in `dispose()`, causing battery drain. Connecting without stopping the scan on Android frequently causes timeouts and adapter hangs.

### Repositories (`app/lib/repositories/partner_repository.dart`)
1. **Offline Mode Overwrites User Configuration**: Throws `TimeoutException` when offline, causing `_config` to be explicitly set to defaults.
2. **Missing `_disposed` Checks Leading to Crashes**: Calling `notifyListeners()` asynchronously without checking `_disposed` crashes the app.
3. **Race Condition in Optimistic UI Updates**: Rapid changes can cause the `finally` block to revert the UI state to a stale version if a subsequent request fails, permanently desynchronizing client and server.
4. **Poor Error Segregation on `.single()`**: New users without config trigger `PGRST116` (no row), treated identically to network errors, loading defaults without upserting.
5. **Incomplete Auth Check in `signUp`**: Returns `true` if `response.user != null`. Bypasses "Confirm Email" requirement, leading to hidden 401 Unauthorized errors.

### Services (`app/lib/services/`)
1. **Critical Architecture Flaw (Main Isolate Suspension)**: Foreground Service in `foreground_service.dart` only keeps the process alive, but Android suspends the main isolate. `Timer` objects in `LocationService` and `BleService` will be throttled/stopped. Must execute in a background isolate.
2. **Geolocator API Breaking Change**: Calls `Geolocator.getCurrentPosition(desiredAccuracy: ...)`, which is deprecated/removed in v13+. Must use `LocationSettings`.
3. **Supabase "Zombie" Subscription Bug**: In `sync_service.dart`, subscriptions can enter a zombie state after a connection drop. Needs explicit lifecycle handling to re-subscribe.
4. **BLE Connection Leak & MTU Timing**: In `ble_service.dart`, rapid disconnects/reconnects without awaiting cleanup can permanently lock the Android BLE stack. Also, requesting MTU immediately upon connection can drop connections.
5. **Android 14+ Foreground Service Types Exception**: `location` and `connectedDevice` types must be explicitly declared in `AndroidManifest.xml` or `flutter_foreground_task` will crash.
6. **MTU Negotiation Race Condition**: Calling `requestMtu` without `await`ing before subscribing to characteristics causes silent failures and MTU = 0.
7. **Reconnection Budget Exhaustion**: `_reconnectAttempt` increments on failure but is never reset on success. Max 3 disconnects per app lifecycle before the service completely dies.

### Models (`app/lib/models/`)
1. **Hard Crash on Null Coordinates**: Strict casting `(json['latitude'] as num).toDouble()` throws a TypeError if Supabase returns `null`.
2. **Timezone Offset Bug in `age` Calculation**: Dart's `DateTime.parse()` interprets timestamps without time zones as Local Time, breaking `age` calculation relative to Supabase's UTC `now()`.
3. **Exception on Strict Numeric Casts**: Strict integer casting `json['double_flick_window_ms'] as int?` throws if Supabase uses a `numeric` column.
4. **Missing `updated_at` in UPSERT payload**: Omits `updatedAt`, meaning it will not auto-update unless a PostgreSQL trigger exists, breaking age logic.
5. **Fragile Hex Color Parsing**: `parseColor(String hexColor)` strictly enforces `length == 8`. Standard 6-character hex or hashes silently fail and fall back to white.

## Deep Scan Findings (Second Pass)

### App Screens (`app/lib/screens`)
1. **BLE Command Flooding / Race Conditions via Sliders**: `Slider.onChanged` immediately executes BLE writes and Supabase API calls. Dragging the slider will emit dozens of concurrent BLE writes per second, guaranteeing GATT `BUSY` errors and DB race conditions. Fix: Use `Slider.onChangeEnd`.
2. **Dangling Watch State (Uncancelled Calibration)**: If a user starts Auto Calibration and presses Back, `dispose()` is executed but `ble.cancelCalibration()` is **never** invoked. The watch remains stuck in calibration mode.
3. **Global BLE Callback Overwrite / Desync**: Screens assign `ble.onCalibrationProgress = _onCalibProgress` in `initState()` and blindly set it to `null` in `dispose()`. Destroying one screen will silently kill the callback for any other active listeners.
4. **Unhandled `RangeError` Exception on BLE IDs**: `bleService.connectedDeviceId?.substring(0, 8) ?? "?"` throws a fatal `RangeError: Value not in range: 8` if the ID is shorter than 8 characters, crashing the UI.
5. **Unhandled `FormatException` in Color Picker**: `WatchConfig.parseColor('FF$currentRgb')` throws a `FormatException` if the stored config is corrupted, empty, or malformed, crashing the app before the dialog renders.
6. **Orphaned Animation Controller Leak**: `_scanAnimController.repeat()` is called, but never attached to a widget, and `stop()` is never called when scanning ends. It leaks animation ticks in the background forever.

### App Build & Config (`app/android`)
1. **State Desync / Unresolved Kotlin Reference**: `android.builtInKotlin=false` is set in `gradle.properties`, but `id("org.jetbrains.kotlin.android")` is missing from `app/build.gradle.kts`. This will crash with `unresolved reference: kotlin`.
2. **Circular Evaluation Dependency**: `subprojects { project.evaluationDependsOn(":app") }` inside `:app` triggers a hard `Circular evaluation dependency` crash in modern Gradle.
3. **Unhandled Exception Risk in `settings.gradle.kts`**: `file("local.properties").inputStream().use {...}` assumes the file exists. If absent (e.g. CI/CD), it causes a `FileNotFoundException` and fails synchronization.
4. **Extreme Memory Over-Allocation**: JVM arguments request `-Xmx8G -XX:MaxMetaspaceSize=4G`. Reserving over 12.5GB of RAM for the Gradle daemon on average systems will cause severe thrashing and OS OOM kills.

### App Widgets (`app/lib/widgets/watch_preview_widget.dart`)
1. **Clock & Distance Rendered Backwards**: `_physicalIndex(i)` makes them render counter-clockwise. `_drawRadarMode` bypasses this, causing modes to be visually desynchronized.
2. **Distance Mode - Max Distance LED Unreachable**: For max distance, `ledsFloat = 12.0`, `partialBrightness = 0.0`. The 12th LED evaluates to `0.0` brightness and never turns on.
3. **Distance Mode - Watch Face Blank at Minimum Distance**: For close distances, `normalized` is `0.0`. The 1st LED evaluates to `0.0` brightness, causing the watch face to go completely dark.
4. **Unhandled Exceptions on NaN**: If `bearing` or `distanceKm` is `NaN`, IEEE 754 logic bypasses safeguards and `double.floor()` throws an `UnsupportedError`, crashing the `CustomPainter` and UI.
5. **Radar Mode LED Interpolation Jitter**: The trailing LED's brightness drops to 50%, then instantly snaps to 0% when the index rolls over, creating a stutter.
6. **Hour Hand Complete Masking**: Overlapping hour and minute hands completely swallow the hour hand instead of blending colors.

### App Models (`app/lib/models`)
1. **State Desynchronization (Clock Skew) in `LocationModel.age`**: Calculates `DateTime.now().difference(updatedAt)` using local device time against Supabase server time. Susceptible to clock skew, leading to negative age (falsely fresh) or falsely stale locations.
2. **Brittle Color Parsing with Silent Failure**: `WatchConfig.parseColor` enforces `hexColor.length == 8`. Anything else silently fails and falls back to solid white.
3. **Unhandled `TypeError` Exceptions during Deserialization**:
   - `LocationModel.fromJson`: `(json['latitude'] as num).toDouble()` throws `TypeError` if latitude is `null` (e.g., empty initial user row).
   - `WatchConfig.fromJson`: Casting `json['haptic_pattern'] as String?` throws if the DB returns an integer `1`.
4. **Unhandled `FormatException` in Date Parsing**: `DateTime.parse(json['updated_at'] as String)` throws if `updated_at` is empty `""`.

### App Repositories (`app/lib/repositories/partner_repository.dart`)
1. **Critical State Desync (`loadConfig` / `saveConfig` Race Condition)**: If `saveConfig` completes while `loadConfig` is still waiting, `loadConfig` blindly overwrites `_config` with stale fetched data because `_pendingRequests` becomes 0.
2. **Data Loss on Concurrent `saveConfig` Failures**: Two rapid requests: R1 succeeds, skips updating `_lastConfirmedServerConfig`. R2 fails, reverts `_config` to `_lastConfirmedServerConfig`. R1's successful change is lost.
3. **Semantic Confusion & Logic Bug in `_isPaired`**: `initialize` treats `_isPaired` as "Partner Paired", but `saveDeviceId`/`clearPairing` treat it as "BLE Device Paired". Connecting/disconnecting the watch corrupts the partner pairing state.
4. **Unhandled Exceptions / Memory Leaks via `notifyListeners`**: `notifyListeners()` is called after async operations without checking `!_disposed`, causing fatal errors if the repository is disposed.
5. **Race Condition & Lost Updates in `updateConfig`**: Concurrent UI calls while `_config == null` both apply updates to the same base config, resulting in a lost update instead of composing changes.
6. **Unhandled Exception Risk in `signOut`**: `await _client.auth.signOut();` lacks `try/catch`. Offline devices will throw an unhandled exception, trapping the user in a broken logout state.

### App Services (`app/lib/services/`)
1. **BLE Infinite Reconnection Loop**: Reconnection limit (`_maxReconnectAttempts = 3`) is completely bypassed because `connectToDevice` immediately calls `disconnect()`, which explicitly resets `_reconnectAttempt` to `0`. It loops forever draining battery.
2. **Concurrent GATT Operations Race Condition**: Proceeding with 5 characteristic subscriptions and a read before `requestMtu` resolves causes the Android BLE stack to silently drop operations or throw `GATT_ERROR 133`.
3. **Unhandled Exceptions on BLE Writes**: Background writes (like `writeBearing`) call `_ble.writeCharacteristicWithResponse` without a try-catch. If a disconnect happens, the unhandled exception crashes the Dart isolate completely.
4. **Overlapping GPS Polling**: Calling `_doGpsUpdate()` (async) and immediately `_scheduleNextUpdate()` causes the scheduled timer to spawn a second concurrent `getCurrentPosition` if the first fix takes longer than 3 seconds.
5. **Fatal Crash on Invalid Config Interval**: `Random().nextInt(remoteMaxS - remoteMinS + 1)` will throw a fatal `RangeError` if the backend sets an interval where `remoteMaxS < remoteMinS`.

### App Initialization (`app/lib/main.dart`)
1. **Memory / Listener Leak**: `partnerRepo.addListener(updateLocationIntervals)` appends without `removeListener`. Retries from the error UI stack up duplicate closures, executing redundant GPS state updates.
2. **State Desync & Silent Bootstrapping Failure**: Catching a bootstrap error unconditionally sets `_initialized = true` bypassing the error UI completely, driving the user to the HomeScreen with broken uninitialized services.
3. **Infinite BLE Feedback Loop**: The app automatically echoes the `onRadarModeChanged` state back to the watch. Since standard BLE notifies on write, this creates a continuous `Watch notifies -> App writes` loop.

### Firmware Overall / Cross-Component Architecture
1. **Split-Brain Wake Threshold Desync**: `wakeThreshold` is duplicated across `RuntimeConfigManager` and `EEPROMManager`. 
    - App JSON update saves to `config.dat` but misses `calib.dat` (reverts on reboot).
    - Threshold-specific characteristic update saves to `calib.dat` but misses `config.dat` (broadcasts old value over JSON).
    - Watch physical calibration updates RAM `runtime_config` but never flushes to `config.dat`.

### Firmware Gestures (`nexus_halo/gesture.cpp` & `gesture.h`)
1. **Dead Code & Logic Contradiction**: The check `flick_count = 1;` is mathematically unreachable due to an earlier timeout logic. Also, `double_flick_window` intended to limit sequence time, but only limits consecutive flick gaps.
2. **Cross-State Cooldown Blinding in `reset()`**: Fails to clear `last_flick_ms`. A flick right after changing state will be ignored due to cooldown check using the old state's timestamp.
3. **"Phantom Flicks" via Data Tearing**: Polling X, Y, Z sequentially creates a race condition with internal IMU sampling. Axes from different time-samples can combine into a massive artificial magnitude. Needs BDU.
4. **Permanent Gesture Lockup via Low Thresholds**: `flick_reset` requires `gyro_mag < threshold * 0.8`. Natural bias/noise can keep it permanently above this line, locking up gesture detection permanently.
5. **Silent Gesture Overwrite**: A multi-flick gesture can be silently overwritten by `GESTURE_TAP_SIMPLE` timeout if `getGesture()` is slightly delayed, completely destroying the recorded multi-flick.
6. **Severe Multi-Flick Latency**: Multi-flicks are not emitted until `double_flick_window` fully expires *after* the gesture, creating massive UI perceived lag.

### Firmware Calibration & Storage (`nexus_halo/eeprom_manager.cpp` & `runtime_config.cpp`)
1. **Atomicity / Data Loss Race Condition**: `InternalFS.remove()` called right before `InternalFS.rename()` (in both `runtime_config` and `eeprom_manager`) completely breaks LittleFS atomicity. A power failure between lines wipes the entire config.
2. **Meaningless Persistent Timestamp**: Uses `millis() / 1000` (uptime) for persistent timestamp. Rebooting resets it, corrupting chronological validity, and it wraps around after 49.7 days.
3. **Unbounded JSON Integer Parsing**: Negative or large values are parsed successfully but cast blindly to 8-bit/16-bit limits, wrapping around silently (e.g., 150% brightness wraps to ~50% visually).
4. **Fragile JSON Parser Fails on Formatting**: `_findJsonString` expects spaces around keys, failing immediately if the app sends valid JSON formatted with tabs `\t` or newlines `\n`.

### Firmware BLE Integration (`nexus_halo/ble_handler.cpp` & `nexus_halo.ino`)
1. **Missing `CHR_PROPS_NOTIFY` Flag**: The Config characteristic is missing the NOTIFY flag. The Bluefruit stack skips creating the CCCD, and `config_char.notify()` fails completely in the background. The app never gets syncs.
2. **MTU Truncation Bug**: `notifyConfig()` attempts to blast 255 bytes, but MTU negotiation is absent. `notify()` cannot automatically fragment like Long Writes. Any payload > 20 bytes is truncated.
3. **BLE Thread Safety Race Condition**: FreeRTOS `ble_task` parses characteristics and modifies global buffers (`config_json_buf`, etc.) without mutexes/critical sections. The main Arduino loop can easily parse halfway-mutated JSON.
4. **Low-Power Advertising State Desync**: `setLowPowerAdvertising()` returns early if connected. When the device eventually disconnects, it uses `restartOnDisconnect(true)` with the fast, un-updated interval, killing battery during sleep.
5. **Invalid `requestConnectionParameter` Signature**: Passed with only 1 argument instead of 3, resulting in invalid supervision timeouts that the SoftDevice will reject.
6. **Time Sync Wrap-Around Bug**: Calculation `millis() - local_millis` overflows fully every 49.7 days without BLE sync, moving the clock 49.7 days into the past.
7. **BLE Commands Mishandled in Deep Sleep**: The main loop either outright ignores (`calib_start`) or consumes without executing (`ble_calib_start_pending`) commands that come in while the display is asleep.

### Firmware Hardware/Core Edge Cases (`nexus_halo/power.cpp`, `led_controller.cpp`, `compass.cpp`, etc.)
1. **RGBW Protocol Desync (CRITICAL)**: Code assumes GRB (24-bit) for the LED buffer but hardware is GRBW (32-bit). Sending 288 bits instead of 384 completely starves the last 3 LEDs and scrambles the colors of the first 9.
2. **Power Cut Bypasses EasyDMA (`led_controller`)**: `setPower(false)` sets `power_on = false` before `show()`, making `show()` return instantly without issuing a black frame to the DMA. LEDs stay floating.
3. **Distance Mode Blackout (`led_controller`)**: Distances `< 1 km` evaluate `normalized = 0`. No LEDs light up, making the watch appear dead at close distances.
4. **Brightness Percentage Overflow (`led_controller`)**: If percentage > 100 is supplied, it scales up and undergoes overflow truncation on cast to `uint8_t`.
5. **Critical Hardware Mismatch Blind to Tilt (`imu_calibrator`)**: Calibration targets vector magnitude deviations. A tilt shifts gravity between axes but maintains 1.0g magnitude. The calibration logic is completely blind to the actual rotational motion that triggers the hardware wake.
6. **Float Truncation `abs()` vs `fabs()` (`imu_calibrator`)**: Overloading issues can cast `abs()` to integer logic, truncating `0.8f` dynamic acceleration to `0`, breaking calibration permanently.
7. **Wrong PDM Pins Disconnected (`power.cpp`)**: Misidentifies PDM pins. Disconnects `P0.26` (onboard Red LED) and `P0.03` (D1) instead of the actual PDM mic pins, killing the LED and failing to save mic power.
8. **Reversed Sleep Sequence (`power.cpp`)**: `__WFE(); __SEV(); __WFE();` is backwards. The correct ARM Cortex-M sleep sequence is `__SEV(); __WFE(); __WFE();`.
9. **I2C Bus Lockup on Power Cycle (`compass.cpp`)**: Disabling internal pull-ups floats the SDA line, causing noise. Re-enabling them without a clock recovery leaves SDA stuck low, permanently killing I2C until hard reset.
10. **Bypassed Recovery Logic (`compass.cpp`)**: Bus fault returns `false` without incrementing `zero_reading_count`. The `reinitBus` recovery mechanism is bypassed, throwing the code into an infinite loop of failed 100ms timeouts.
11. **Massive CPU Hogging (`compass.cpp`)**: `rreg(STATUS)` polls aggressively using `delayMicroseconds()`, locking the CPU for ~13ms out of every 33ms cycle, stalling animations.
12. **Ghost Vibrations (`haptic.cpp`)**: `stopMotor()` turns off the pin but leaves `pattern_active = true`. The next `update()` instantly turns it back on, or randomly triggers phantom vibrations if it wakes up.
13. **Dropped Haptic Commands (`haptic.cpp`)**: If the app sends a new haptic command while already vibrating, the state machine early-returns and the new pattern is completely lost.
14. **Compass Calibration Silent Fail (`nexus_halo.ino`)**: Calibrates in background of `STATE_CLOCK_CONNECTED`. Standard 5-second sleep timeout interrupts the 10-second calibration window, breaking the offset math.

### Firmware State Machine (`nexus_halo/state_machine.cpp` & `state_machine.h`)
1. **`haptic_rx_pending` Race Condition Aborting Wake-up**: If a BLE haptic event arrives exactly when waking up, it transitions to `STATE_HAPTIC_RX` without updating `previous_state` (which remains `STATE_DEEP_SLEEP`). After 3 seconds, it returns to `STATE_DEEP_SLEEP`, abruptly turning off the screen.
2. **State Desynchronization in `STATE_RADAR_MODE` Timeout**: If BLE drops while in radar mode, it eventually times out and hardcodes a transition to `STATE_CLOCK_CONNECTED` without checking `isConnected()`. This flashes the connected UI for one frame before transitioning to disconnected.
3. **Dropped Background Events due to Same-State Re-entry**: If the app sends multiple `HAPTIC_RX` commands quickly, the second transition is aborted because `new_state == current_state`. The timer is not reset, dropping the second notification completely.

### 11. Hardware vs UI - LED Mirroring Inversion Bug
- **Issue**: Both the physical watch firmware (led_controller.cpp) and the Flutter app UI (watch_preview_widget.dart) featured an index inversion logic snippet ((LED_COUNT - index) % LED_COUNT).
- **Impact**: This caused both the physical watch and the app preview to render the clock hands running counter-clockwise. The two reversed logics masked each other's intent.
- **Resolution**: Stripped inversion math out of both files entirely to restore the default clockwise iteration.
